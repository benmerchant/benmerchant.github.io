Modular Architecture

  • high-level modules: root, component, common
  • root: bootstraps app
  • import other two into root module
  • other two require lower-level componenet modules
    • components, controllers, services, directives, filters, tests
    • common module - non-reusable/app-specific
    • the lower level modules are 'reusable features'
Components (are NOT directives)
  • templates with a controller
  • contain bindings that define
    • inputs/outputs for data/events
    • lifecycle hooks
      • $onChanges is called before $onInit
    • ability to use one-way data flow & event Objects to get data back up to a parent component
      • components recieve data, use '<', one-way databinding syntax
      • NEVER '=' ever again ever
      • component has bindings, use $onChanges to clone one-way binding data to break Objects passing by ref and updating parent data
      • use $event: {} as function arg in parent method (stateful component)
      • pass an $event: () object back up from a stateless component
        • bonus: use an 'EventEmitter' wraper with .value(). avoids manual $event Object creation
  • Properties (approved)
    • bindings - '@', '<', '&' only
    • controller
      • never need one by itself, probably need a 'Stateless Component'
      • use 'constructor' for DI
      • use 'ng-annotate' 'ngInject'; for $inject annotations
      • use arrow functions to access lexical scope OR sometimes it's acceptable to use 'let ctrl = this;'
      • bind all public functions directly to the class
      • use 'require' alongside '$onInit' to ref any inherited logic
  • template
  • templateUrl
  • transclude?!?!?
  • Stateful, Stateless, Routed
    • Stateful aka smart/container Components
      • fetches state: communicating with a backend API through a service
      • does not directly mutate state
      • renders child components that mutate state
      • no directives (like ng-repeat etc) inside the template
      • delegate data & functions into stateless child components
    • Stateless aka dumb/presentational Components
      • defined Inputs & Outputs via bindings: {}
        • Inputs enter through attribute bindings
        • Outputs leave through events
      • mutates state, pass back up on event (click, submit, etc)
      • agnostic regarding source of data
      • highly reusable
      • does not fetch state
        • merely receives it
        • mutates an Object via controller logic associated with it
        • passes Object back to parent component through property bindings
      • clone the initial binding Object with $onChanges and reassign it
        • Object.assign({ }, this.parentObject)
        • so parent data isn't affected until form is submitted (event)
        • alongside one-way dataflow new binding syntax '<'
    • Routed Components
      • basically: stateful component w/ routing definitions
      • define its own routing logic
      • data 'input' done via route resolve (optional, still available in controller via service calls)
      • ui-router resolve properties can directly map across to bindings
      • some (or all idk) are essentiall 'views'
Directives <WebDevJournal_3 p96 d9-jun-2019 extended>
  • DON'T declare templates/controllers
  • DON'T receive data through bindings
  • DO ONLY use to decorate the DOM, extending existing HTML that's already created with .component()
  • DO bind a direcive to a template inside a component custom DOM events/APIs and logic
  • DON'T forget $postLink for sensible amount of DOM manipulation, but DONT just migrate all DOM manipulate there, DO try to use a directive for non-Angular thingsf
  • DONT EVER use templates, scope, bindToController, controllers
  • DO ALWAYS use restrict: A
  • DO use compile & link where necessary
  • DO destroy and unbind event handlers inside $scope.$on('destroy', fn);
